# 用飞书当 CMS：lizizai-blog 的架构演进

大多数博客系统都要回答一个问题：文章存在哪。WordPress 用数据库，Ghost 用数据库，Hexo 用文件。lizizai-blog 的答案是——飞书文档。

这不是噱头。我每天在飞书里写文章，它就是一个我本来就在用的协作工具：富文本、能贴图、能多人看。一个 GitHub Action 定时把文档同步到 Cloudflare R2，博客从 R2 读。我不用维护 CMS，不用备份数据库，内容始终在我自己的飞书账号里。

代价是多了一条"同步链路"——而正是这条链路，后来给我惹出了整个项目最大的麻烦。但这是后话。先说清楚：为什么一个正经博客，要把内容源交给一个办公软件。

## 为什么是飞书

最直接的答案：我本来就在用飞书。

写博客最大的敌人不是技术，是摩擦。WordPress 要登后台、用它那个编辑器、管插件更新；Ghost 好一点，仍是一个"你得去它那里写"的系统；Hexo、Hugo 这类静态生成器，写作退化成编辑 Markdown 文件再 git 提交。每一种都在"想写"和"写下来"之间，插了一道工序。

飞书没有这道工序。我每天打开飞书处理工作，顺手就能写一篇，用的是我熟悉的富文本编辑器。写完也不用"发布"——文档就在那，一个定时任务会把它搬进博客。

但"把内容源交给一个办公软件"是有代价的：我不再直接拥有一堆 Markdown 文件，而是拥有"飞书文档 + 一条同步链路"。这条链路后来给我惹了整个项目最大的麻烦——但那是后面的故事。先看它怎么工作：每天凌晨，一个跑在 GitHub Actions 上的脚本登录飞书，把文档树下载下来，转成 Markdown 和 JSON，写进 Cloudflare R2；博客的 Next.js 从 R2 读这些文件，用 ISR 缓存一小时。

换来了什么？一整个后端没了。没有 CMS 要部署，没有数据库要备份，没有后台要维护。内容所有权也还在我这——文档在我的飞书账号里，随时能导出。

## 当同步链路崩掉

这条链路看起来人畜无害，直到有一天它崩了。

崩在 Cloudflare Workers 的一个限制上：子请求数。一个 Worker 单次执行能发出的外部请求（fetch）是有上限的。同步几十篇飞书文档，每篇要列目录、下载内容、拉图片、写 R2——请求量轻松破限。同步任务第一次大规模跑起来时，直接报错中断，R2 里只同步了一半文章。

接下来的两天，我试了所有"在 Workers 里绕过去"的办法，每一个都像在解一道出题人不让你解的题：

- **增量同步**：只同步有变化的文档，减少请求量。有用，但文档一多还是超。
- **全局索引预加载**：先把目录结构一次性拉下来缓存，减少重复列目录。省了一些，还是超。
- **Cloudflare Queue 分片**：把任务切成小块丢进消息队列分批处理。结果撞上 Queue 免费计划的另一道墙。
- **Worker 自调用分批**：让 Worker 处理完一批后 HTTP 调用自己处理下一批。听起来优雅，实测 workers.dev 自调用会超时，链子断在中途。

每个方案都解决了一部分问题，又制造出新的。两天下来我意识到一件事：我一直在试图"绕过"一个平台的硬限制，而这个任务——批量下载、转换、上传几十个文件——本来就不适合塞进一个有执行时长和子请求上限的无服务器函数里。

于是我换了平台。同步逻辑整个搬出 Cloudflare，交给 GitHub Actions：一个定时触发的 Node.js 脚本，没有子请求限制，没有执行时长焦虑，跑多久都行。原来的 Worker 没删，降级成一个只读服务——只保留健康检查、同步状态查询、飞书目录调试三个端点。

回过头看，这是个比"绕过限制"清楚得多的判断。Workers 适合轻量、短时、高并发的边缘计算；批量同步是重 I/O、长耗时、单次跑的任务，天生属于一个能跑几分钟的环境。问题不在于 Workers 限制太严，而在于我把错误的任务放进了错误的平台。当"绕过"的复杂度超过"换平台"的成本时，答案就该是换平台。

## 拼出来的互动层

如果说"用飞书当 CMS"省掉了一个内容后端，那博客剩下的功能——评论、点赞、浏览量、邮件订阅、搜索——则是用另一种方式省掉的：每一个都接一个现成的 Cloudflare 服务，自己一行子系统代码都不写。

这不是图省事（虽然也省事）。每一个功能单独看都不难写，但加起来是一个完整的互动层：要存数据、要做接口、要防滥用、要监控。一个独立开发者维护五个这样的子系统，光是保证它们不挂、不丢数据、不被人刷，就够喝一壶。把它们外包给五个专门做这件事的服务，等于把"运行一个互动层的成本"分包了出去。

接进来的是这些：

| 功能 | 服务 | 数据落在 |
| --- | --- | --- |
| 评论 | cf-comment | Cloudflare D1 |
| 点赞 | emaction | Cloudflare D1 |
| 浏览量 | Webviso | Cloudflare D1 |
| 站点分析 | counterscale | Cloudflare D1 |
| 邮件订阅 | Resend | — |
| 搜索 | Pagefind | 构建时静态索引 |

有意思的是前四个全是 Cloudflare 生态——数据都存在 D1 上，只是各跑各的 Worker。这带来一个意外好处：博客这边不需要为每个服务写一套失败处理，只要在调用处加一个开关。点赞服务挂了，`isEmactionEnabled()` 返回 false，按钮不显示，文章照常读；浏览量服务挂了，那次请求拿不到数，列表照常渲染。没有一个外部服务的故障能把整篇文章拖崩。

聚合发生在数据层。打开一篇文章时，`getArticleBySlug` 会并行去捞点赞、浏览量、评论数——三个请求走 `Promise.all`，哪个慢或失败都不阻塞其他。首页列表更克制：只批量拿一次浏览量（`getBatchViews`），不逐篇请求，这是后来首页编译从 2N+2 次请求压到 2 次的关键。

这种拼装不是没有代价。代价是你把博客的"互动真实性"托付给了第三方服务，任何一个停服或改价，你都得跟着动。但权衡下来，这比自建五个子系统、再为每个兜底，要轻得多。

## ISR 是一组旋钮，不是一个开关

ISR 不是一个开关，是一组分层的旋钮。

Next.js 的增量静态再生（ISR）让页面在构建后仍能定期更新，但"定期"是多久，得自己定。lizizai-blog 没有一刀切，而是按每种数据的"鲜度需求"给了不同的 revalidate：文章内容和列表 3600 秒（一小时），浏览量 300 秒（五分钟），点赞 60 秒（一分钟），评论则根本不进缓存，每次实时去取。

这个分层背后是一个简单的问题：这份数据多久会变一次？文章一天同步一次，缓存一小时足够，用户感知不到延迟；浏览量每分钟都在涨，但不能太实时（否则每个访客都打一次 Webviso），五分钟是个折中；点赞是强互动信号，用户点完想立刻看到数字跳，一分钟勉强够用；评论数量同理，但评论频率低、实时性强，干脆不缓存。

把这些数字摆在一起，能看到一种"按变更频率对齐缓存时长"的纪律：

- `revalidate: 3600` — 文章 / 列表 / 分类（内容，日级变更）
- `revalidate: 300` — 浏览量（分钟级变更，五分钟折中）
- `revalidate: 60` — 点赞（强互动，分钟内期望反馈）
- 实时 fetch — 评论数（低频但要求新鲜）

这套分层和上一节的"按需聚合"是同一套思路：不为所有数据付同样的代价。变化慢的，给 ISR 充分缓存；变化快的，缩短缓存；必须新鲜的，放弃缓存直接取。博客用一行 `next: { revalidate }` 把这个判断写进每一个 fetch，剩下的交给框架。

## 留一条本地退路

博客的数据访问其实有两套。生产用 `lib/blog-data.ts`，从 R2 读；旁边还躺着 `lib/content.ts`，从本地 `content/` 目录读 MDX 和 YAML。后者的注释写着"被 blog-data.ts 替代"，全项目 grep 下来，已经没有任何地方 import 它。

那为什么不删？

因为它是一条退路。R2 是个外部服务，会抽风、会限流、会因为我手滑改错配置而整站读不到数据——这些我都遇到过。当 R2 出问题时，本地 MDX 数据层是一个可以立刻切回去的备份：文章以文件形式存在仓库里，开发时不用联网，调试时改一个文件立刻生效。保留它的成本几乎为零（代码已经在那），收益是"生产挂了，我还有个能跑的本地版"。

这是一类很典型的取舍：生产数据层为了性能和规模上云，但留一个不依赖云的影子版本兜底。两套代码之间隔着一行 import 的距离，哪天 R2 真的靠不住，切换的成本是改一个路径常量，不是重写一个数据层。

## 一篇文章的四种形态

飞书当 CMS 还带来一个意外收获：一篇文章不再只是"一段文字"。

lizizai-blog 的文章有四种内容类型——普通文章、播客、幻灯片、HTML。这不是四个独立系统，而是同一套同步链路里，飞书文件夹的不同结构映射成不同的渲染方式。一个播客文章，飞书里是文章正文加一个装着音频、封面、脚本的子文件夹；同步时 Worker 把它们归组，前端拿到一个播客列表，每项是音频加脚本的组合。幻灯片类似，飞书里是截图序列，同步成 slides.json。HTML 类型更直接——整篇就是手写的 HTML，原样存进 R2，用 iframe 嵌进文章页。

这里面踩过坑。播客最早是"一篇文章一个播客"的模型，后来发现一篇文章可能配多期播客，于是把单播客重构成多播客列表——为了不让重构变成黑盒，那次我第一次给这个项目写了单元测试，把文件归组的纯函数从同步逻辑里抽出来单独测。一个内容模型的调整，逼出了项目的第一套测试。

数据层处理这些类型时很克制：`getArticleBySlug` 拿到文章后，根据 contentType 决定还要不要去拉 slides.json、解析播客列表、构造 HTML 的 R2 地址。新格式优先，旧格式兼容（比如早期播客固定叫 audio.mp3，现在还认这个路径）。类型多，但分支清楚。

## 这一切是怎么来的

讲到这里，博客像是被一次性设计成这样的——飞书当源、R2 当存、Worker 当同步、边缘服务当功能。但其实没有一次设计。

它的第一次提交是一个克隆。2025 年 10 月，项目从一个叫 "Letters" 的模板 Fork 过来，后端是自托管的 Strapi，数据库是 Supabase，带完整的用户登录、订阅、评论——一个标准的内容社区。然后第一件事是往回砍：移除用户认证，清掉 UI 里所有用户相关的元素，换成一个匿名的访客身份系统（每个访客自动分一个武侠风格的昵称和头像，不用注册）。理由很简单：一个人写博客，不需要用户系统，也不想让评论依赖登录。

砍完之后项目沉寂了几个月。再动是在 2026 年 4 月，那次带来的是真正定义这个博客的决定：把内容源从 Strapi 换成飞书。Strapi 是个好 CMS，但它要部署、要运维、要备份数据库，而我写文章的工具本来就是飞书——与其维护一个 CMS 去复制飞书里已有的内容，不如直接从飞书读。于是有了同步链路，有了 R2，有了后面所有的取舍。

回头看，这个博客的架构不是某一天画在白板上的，是一连串"这个我不需要""那个我用现成的"累积出来的。每一次转向都在丢东西——丢掉用户系统、丢掉自建 CMS、丢掉自建互动层——丢到最后，剩下的恰好是"一个写作者真正需要"的部分。

## 博客开始生产自己的工具

故事到这还没完。这个用飞书当 CMS、用别人的服务拼出功能的博客，后来长出了一个奇怪的能力：它开始生产自己的工具。

最先长出来的是 `lizizai-html`——一个 skill，用博客自己的设计系统，生成可以嵌进文章的独立 HTML 页面（比如一篇 AI 日报）。主题样式被抽出来部署到 R2 CDN 上，任何生成的 HTML 引一下就能和博客视觉一致。然后是 `ai-daily-extract`，把结构多变的 AI 日报内容，用一套 JSON Schema 加一个 LLM 提取脚本，变成统一的结构化数据。再后来是配图：一个叫 ian-xiaohei 的插图 skill，给文章里的抽象概念画怪诞的小黑配图——这篇博客里那些插图，就是它画的。

这有点递归的意思：博客的内容生产流水线——模板、提取、配图、渲染——被一件件抽象成可复用的工具，而这些工具反过来又在给博客生产内容。写博客的人原本在用工具，写着写着，工具本身成了博客的一部分。

回到开头那个问题：文章存在哪？lizizai-blog 的答案是飞书。但更完整的答案是：存在飞书，同步到 R2，用别人的服务拼出互动，按鲜度分层缓存，留一条本地退路——然后，把这套流程里值得复用的部分，再变成工具，喂给下一篇。这不是一个博客系统的设计，是一个写作者和工具共同进化的过程。
